## 📖 Key Concepts

### 1. Typical Backend Project Structure

```
my-project/
├── src/                    # Source code
│   ├── api/               # API routes (v1, v2)
│   ├── config/            # Configuration files
│   ├── middleware/        # Request interceptors
│   ├── repositories/      # Database access
│   ├── services/          # Business logic
│   ├── types/             # TypeScript interfaces
│   └── index.ts           # Entry point
├── package.json           # Project info & dependencies
├── tsconfig.json          # TypeScript configuration
├── .env                   # Environment variables (secrets)
└── node_modules/          # Installed packages (auto-generated)
```

### 2. What is package.json?
The "ID card" of your project. Contains:
- Project name and version
- Scripts (commands to run)
- Dependencies (packages your project needs)

```json
{
  "name": "my-project",
  "version": "1.0.0",
  "scripts": {
    "start": "node dist/index.js",
    "dev": "ts-node src/index.ts",
    "build": "tsc"
  },
  "dependencies": {
    "fastify": "^4.0.0"
  },
  "devDependencies": {
    "typescript": "^5.0.0"
  }
}
```

### 3. Dependencies vs DevDependencies

| Type | Purpose | Example |
|------|---------|---------|
| dependencies | Needed to RUN the app | fastify, bcrypt, jsonwebtoken |
| devDependencies | Needed only for DEVELOPMENT | typescript, jest, eslint |

### 4. What is npm?
**npm** = Node Package Manager
- Downloads and installs packages
- Manages project dependencies
- Runs scripts defined in package.json

Common commands:
```bash
npm init              # Create new package.json
npm install           # Install all dependencies
npm install fastify   # Install specific package
npm run dev           # Run the "dev" script
npm run build         # Run the "build" script
```

### 5. Understanding Version Numbers
Version format: `MAJOR.MINOR.PATCH` (e.g., `4.2.1`)

- **MAJOR** (4): Breaking changes
- **MINOR** (2): New features, backward compatible
- **PATCH** (1): Bug fixes

Symbols in package.json:
- `^4.2.1` - Accept minor updates (4.x.x)
- `~4.2.1` - Accept patch updates (4.2.x)
- `4.2.1` - Exact version only

### 6. The node_modules Folder
- Contains all installed packages
- Auto-generated by `npm install`
- **Never commit to git** (add to .gitignore)
- Can be deleted and recreated anytime

**src/services/userService.ts**:
```typescript
// Business logic for users
import { User, CreateUserInput } from "../types/user";

// Simulated database
let users: User[] = [];
let nextId = 1;

export function createUser(input: CreateUserInput): User {
    const newUser: User = {
        id: nextId++,
        name: input.name,
        email: input.email,
        createdAt: new Date()
    };
    users.push(newUser);
    return newUser;
}

export function getAllUsers(): User[] {
    return users;
}

export function getUserById(id: number): User | undefined {
    return users.find(user => user.id === id);
}
```

**src/index.ts**:
```typescript
// Entry point - where the application starts
import { config } from "./config";

console.log(`Environment: ${config.environment}`);

console.log("\nUser with ID 1:", getUserById(1));
```

## 💬 Q&A Session Notes

### Q: What do import and export mean?

**A:** **export** = make something in this file available for other files to use. **import** = use something that another file exported. So export = "can be used outside"; import = "use from outside".

---

### Q: Why does id start at 1 with nextId++?

**A:** `nextId++` is post-increment: it returns the current value (1), then increments. So first user gets id: 1, then nextId becomes 2. If you used `++nextId` the first id would be 2.

---

### Q: Why const newUser instead of let?

**A:** We never reassign `newUser` - we create one object and keep that reference. const = cannot reassign; the object can still be mutated. So const is correct and preferred.

---

### Q: Can we use node instead of ts-node to run index.ts?

**A:** No. Node runs only JavaScript (.js). TypeScript (.ts) must be run with ts-node (or compile with tsc then run node dist/index.js). So keep `"start": "ts-node src/index.ts"` or use build + node.

---

### Q: Is npm run start the same as npx ts-node src/index.ts?

**A:** Yes, when run from the project root. Both run `ts-node src/index.ts`. Use the same path (e.g. src/index.ts) when running manually.

---

### Q: Can we use a custom script name like "runapp"?

**A:** Yes. The script name (key) is up to you: "runapp", "dev", "go". The value must be a real command (e.g. ts-node). So `"runapp": "ts-node src/index.ts"` then `npm run runapp`.

---

### Q: dependencies vs devDependencies - can we use only one?

**A:** Runtime packages must be in dependencies (production installs skip devDependencies). Dev-only tools go in devDependencies. You can't put everything only in devDependencies or production would miss runtime libs.

---

### Q: Where do we run npm install?

**A:** In the folder that contains package.json (the project root). npm reads that package.json and creates node_modules there.

---

### Q: Should we commit node_modules?

**A:** No. Don't commit node_modules: it's large, regenerable with npm install, and can be platform-specific. Add node_modules/ to .gitignore. Do commit package.json and package-lock.json.

---

### Q: What does -D mean in npm install -D typescript?

**A:** -D (or --save-dev) means "add this package to devDependencies". So -D = devDependencies. No -D = dependencies.

---

### Q: Is package-lock.json the same as node_modules?

**A:** No. node_modules = folder with the installed package code. package-lock.json = file that locks exact versions. npm install creates both. Commit package-lock.json; don't commit node_modules.

---

### Q: What does port 3000 mean? localhost:3000 or using port 3000 for the API?

**A:** Both are correct. `port: 3000` means the backend listens on port 3000, so you use port 3000 for this API and you access it at **http://localhost:3000**. localhost = your machine; 3000 = the port. When you add a server (e.g. Fastify), you do `app.listen({ port: config.port })` and then the API is reachable at localhost:3000.
